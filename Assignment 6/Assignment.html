Visualizing/Simulating a pipelined processor execution

COP 290: Design Practices           
Assignment 5: Visualizing/Simulating a pipelined processor execution

Problem: Given the assembly code, highlight the relevant control signal/datapath on a pipelined processor SVG image. Further, your simulator should be able to give register/memory dump and cycle count.

Tools to use: Flex, Bison

The assignment is divided into two parts (for ease of explanation). You will have only one assignment deadline.


PART 1 : DRAWING AN SVG IMAGE & THREADING

  • SVG Image:

Draw an SVG image of processor (with hazard detection and forwarding unit). Refer to fig. 4.65: The final datapath and control given in Patterson, David A., and John L. Hennessy. Computer organization and design: the hardware/software interface. Newnes, 2013. (5th edition) Here

Some details are missing. Please fill the missing details from fig. 4.57 and fig. 4.51 and create a single SVG image with with hazard detection and forwarding unit.  Consider the MIP’s program:

In the figure below, appropriate pipelining stages have been highlighted for the LW and SUB instruction. Rest of the instructions are omitted for simplicity.

Following is the list of supported instructions (for part1):

  1. The memory-reference instructions load word ( lw ) and store word ( sw )

  2. The arithmetic-logical instructions add , sub , AND , OR , and slt

  3. The instructions branch equal ( beq ) and jump (j)


Refer “A.10 MIPS R2000 Assembly Language” for the instruction formats. Some of the instruction formats are given below:


  1. Load Word


  1. Add

  1. AND


  • Threading:

This should be a multi threaded program. Each "stage" should be handled by a "thread". So, in the steady state, there must at least 6 threads (5 stages, 1 display) which would be in place.  You have to also display in one corner of the output which thread is active and is working. On multi cores, we should be able to see more than one thread working in parallel. Use pthread library to implement threading.


Input :

input.hex: HEX file (containing list of instructions in hexadecimal format).

To do :                                           

Read input.hex. In case hex is incorrect format, do error handling and then gracefully exit. If the hex is in correct format proceed. From NOW, please assume that in case you are taking user input, you need to DO error handling (by default).

On each click of ENTER, advance to the next timestep. Highlight the appropriate datapath and control signal for each pipeline stage. Mention the instruction on the top of each pipeline stage.

Further, display the active threads in one corner.

Output :               

out.svg: SVG (displayable in chrome browser)

Processor simulator: Design a processor simulator, which reports, the time to execute a program and further gives a dump of (the contents) register file and memory. Assume each functional unit (to access) takes one cycle. The memory/register file get updated in the respective write back stages. You can take a look at MARS (supported for windows only)

Deadline :               

Sunday, 26 March 2017,10:00 PM


Accepting file input (Part1)

Your program should accept file input and should generate the result in a file. The command for running your program:

processor_simulator <input_HEX_file> <output_SVG_file>


Accepting file input (Part2)

Your program should accept file input and should generate the result in a file. The command for running your program:

processor_simulator <input_HEX_file> <output_SVG_file> <output_results_file>

Format of the results file will be updated.
